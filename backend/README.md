# Архитектура Backend (Spring Boot)

Данный проект представляет собой серверную часть ChatChat, реализованную на базе Java и фреймворка Spring Boot. 
Ниже приведено описание основных пакетов (директорий внутри `src/main/java/com/example/chat`) и их зон ответственности.

## Описание пакетов

*   **`config`**
    Пакет отвечает за регистрацию настроек приложения.
    *   `WebMvcConfig` — конфигурация веб-слоя (настройка CORS, чтобы к API мог обращаться фронтенд, а также обработка статических ресурсов, например, папки `uploads`).
    *   `WebSocketConfig` — конфигурация WebSocket-соединений, необходимых для двустороннего обмена сообщениями в реальном времени (поверх HTTP).

*   **`controller`**
    Слой контроллеров (Endpoints). Отвечает за прием HTTP-запросов от клиента (фронтенда) и возврат HTTP-ответов.
    *   `AuthController` — регистрация и авторизация.
    *   `ChatController` — управление чатами (создание, получение списка, приглашения и т.д.).
    *   `FileController` — загрузка медиа и файлов.

*   **`dto`** (Data Transfer Object)
    Объекты передачи данных. Помогают разделить структуру базы данных (Entity) и структуру данных, которые приложение принимает или отдает клиенту по API. Сюда входят входящие запросы (например, логин/пароль) и ответы.

*   **`entity`**
    Модели данных слоя базы. Это JPA-сущности (Hibernate), которые напрямую маппятся в таблицы базы данных (например, `User`, `Message`, `ChatRoom`, `RoomInvitation`).

*   **`repository`**
    Слой доступа к данным (Data Access). Содержит интерфейсы `Spring Data JPA` (например, `UserRepository`, `MessageRepository`), выступающие прослойкой между `service` и базой данных. Spring автоматически реализует их методы для CRUD (Create, Read, Update, Delete).

*   **`service`**
    Слой бизнес-логики. Здесь происходят главные вычисления и операции.
    *   Сервисы (такие как `UserService`, `ChatService`, `MessageService` и `FileStorageService`) содержат логику работы приложения.
    *   `ChatWebSocketHandler` — специальный класс для обработки "сырых" WebSocket событий (подключение, текстовые сообщения, отключение клиента).
    *   `JwtHandshakeInterceptor` — перехватчик для проверки безопасности (JWT-токена) при установке WebSocket-соединения.

*   **`exception`**
    Содержит классы для обработки ошибок.
    *   `RestExceptionHandler` — глобальный перехватчик исключений (ControllerAdvice), который превращает ошибки Java в понятные JSON-ответы с правильными HTTP статусами.

*   **`util`**
    Вспомогательные (утилитные) классы, например `AuthTokens`, для переиспользуемых мелких задач.

---

## Как слои взаимодействуют между собой

Архитектура построена по классическому паттерну **MVC (Model-View-Controller)** применительно к REST API:

1.  **Поток REST-запроса:**
    *   **Фронтенд** отправляет HTTP-запрос (например, создать чат).
    *   Запрос принимает **Controller** (`ChatController`). Он валидирует `DTO` и вызывает нужный метод из **Service**.
    *   **Service** (`ChatService`) реализует логику (проверяет права пользователя и т.д.), создает **Entity** объекта и передает его в **Repository**.
    *   **Repository** (`ChatRoomRepository`) сохраняет или извлекает данные напрямую из базы данных.
    *   После этого **Service** возвращает результат в **Controller**.
    *   **Controller** заново конвертирует его в `DTO` и отдает клиенту в формате JSON.

2.  **Поток реального времени (WebSocket):**
    *   После авторизации **Фронтенд** открывает WebSocket соединение с `/api/ws/chat`.
    *   В момент рукопожатия (Handshake) срабатывает `JwtHandshakeInterceptor`, проверяя токен в URL пользователя.
    *   Соединение переходит к `ChatWebSocketHandler`. Когда пользователь отправляет сообщение в чат:
        1. Хендлер принимает JSON.
        2. Разбирает его на тип события и делегирует `MessageService` / `ChatService`.
        3. Сообщение сохраняется в БД.
        4. Хендлер пересылает сообщение нужным адресатам (другим участникам комнаты или прямым собеседникам), чьи WebSocket-сессии сохранены в памяти сервера.
